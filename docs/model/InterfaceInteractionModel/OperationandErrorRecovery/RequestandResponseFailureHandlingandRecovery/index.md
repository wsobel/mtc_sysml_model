# Request and Response Failure Handling and Recovery Package

A significant feature of the <span class="hoverterm">request and response<span class="definition" data-term="request and response">communications pattern that supports the transfer of information between an *agent* and a *client*. </span></span> <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> is the ability for either piece of equipment to detect a failure associated with either the <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> or <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> actions.  When either a failure or unexpected action occurs, the <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> and the <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> portion of the <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> can announce a `FAIL` state upon detecting a problem.  The following are graphical models describing multiple scenarios where either the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> or <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> detects and reacts to a failure.  In these examples, either the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> or <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> announces the detection of a failure by setting either the <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> or the <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state to `FAIL`.  

Once a failure is detected, the <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> provides information from each piece of equipment as they attempt to recover from a failure, reset all of their functions associated with the <span class="hoverterm">interface<span class="definition" data-term="interface">means by which communication is achieved between independent systems.</span></span> to their original state, and return to normal operation.

The following sections are scenarios that describe how pieces of equipment may react to different types of failures and how they indicate when they are again ready to request a service or respond to a request for service after recovering from those failures:

### Responder Fails Immediately

In this scenario, a failure is detected by the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> immediately after a <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> for service has been initiated by the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span>.

![Responder Fails Immediately]({% link figures/mtconnect.png %} "Responder Fails Immediately"){: width="80.0%"}

In this case, the <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> transitions to `ACTIVE` and the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> immediately detects a failure before it can transition the <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state to `ACTIVE`.  When this occurs, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> transitions the <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state to `FAIL`.

After detecting that the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> has transitioned its state to `FAIL`, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> **MUST** change its state to `FAIL`.

The <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span>, as part of clearing a failure, resets any partial actions that were initiated and attempts to return to a condition where it is again ready to request a service.  If the recovery is successful, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> changes its state from `FAIL` to `READY`.  If for some reason the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> cannot return to a condition where it is again ready to request a service, it transitions its state from `FAIL` to `NOT_READY`.

The <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span>, as part of clearing a failure, resets any partial actions that were initiated and attempts to return to a condition where it is again ready to perform a service.  If the recovery is successful, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> changes its <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state from `FAIL` to `READY`.  If for some reason the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> is not again prepared to perform a service, it transitions its state from `FAIL` to `NOT_READY`.

### Responder Fails While Providing a Service

This is the most common failure scenario.  In this case, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> will begin the actions required to provide a service.  During these actions, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> detects a failure and transitions its <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state to `FAIL`.

![Responder Fails While Providing a Service]({% link figures/mtconnect.png %} "Responder Fails While Providing a Service"){: width="80.0%"}

When a <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> detects a failure of a <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span>, it transitions it state from `ACTIVE` to `FAIL`.  

The <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> resets any partial actions that were initiated and attempts to return to a condition where it is again ready to request a service.  If the recovery is successful, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> changes its state from `FAIL` to `READY` if the failure has been cleared and it is again prepared to request another service.  If for some reason the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> cannot return to a condition where it is again ready to request a service, it transitions its state from `FAIL` to `NOT_READY`.

The <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span>, as part of clearing a failure, resets any partial actions that were initiated and attempts to return to a condition where it is again ready to perform a service.  If the recovery is successful, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> changes its <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state from `FAIL` to `READY` if it is again prepared to perform a service.  If for some reason the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> is not again prepared to perform a service, it transitions its state from `FAIL` to `NOT_READY`.

### Requester Failure During a Service Request

In this scenario, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> will begin the actions required to provide a service.  During these actions, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> detects a failure and transitions its <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> state to `FAIL`.

When the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> detects that the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> has transitioned its <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> state to `FAIL`, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> also transitions its <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state to `FAIL`. 

The <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span>, as part of clearing a failure, resets any partial actions that were initiated and attempts to return to a condition where it is again ready to request a service.  If the recovery is successful, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> changes its state from `FAIL` to `READY`.  If for some reason the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> cannot return to a condition where it is again ready to request a service, it transitions its state from `FAIL` to `NOT_READY`.

The <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span>, as part of clearing a failure, resets any partial actions that were initiated and attempts to return to a condition where it is again ready to perform a service.  If the recovery is successful, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> changes its <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state from `FAIL` to `READY`.  If for some reason the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> is not again prepared to perform a service, it transitions its state from `FAIL` to `NOT_READY`.

### Requester Changes to an Unexpected State While Responder is Providing a Service

In some cases, a <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> may transition to an unexpected state after it has initiated a <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> for service.  

As demonstrated in *Figure Requester Makes Unexpected State Change*, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> has initiated a <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> for service and its <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> state has been changed to `ACTIVE`.  The <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> begins the actions required to provide the service.  During these actions, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> transitions its <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> state back to `READY` before the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> can complete its actions.  This **SHOULD** be regarded as a failure of the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span>.

![Requester Makes Unexpected State Change]({% link figures/mtconnect.png %} "Requester Makes Unexpected State Change"){: width="80.0%"}

In this case, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> reacts to this change of state of the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> in the same way as though the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> had transitioned its <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> state to `FAIL` (i.e., the same as in Scenario 3 above).

At this point, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> then transitions its <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state to `FAIL`.

The <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> resets any partial actions that were initiated and attempts to return to its original condition where it is again ready to perform a service.  If the recovery is successful, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> changes its <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state from `FAIL` to `READY`.  If for some reason the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> is not again prepared to perform a service, it transitions its state from `FAIL` to `NOT_READY`.

> Note: The same scenario exists if the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> transitions its <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> state to `NOT_READY`.  However, in this case, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> then transitions its <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> state to `READY` after it resets all of its functions back to a condition where it is again prepared to make a <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> for service.


### Responder Changes to an Unexpected State While Providing a Service

Similar to Scenario 5, a <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> may transition to an unexpected state while providing a service.  

As demonstrated in *Figure Responder Makes Unexpected State Change*, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> is performing the actions to provide a service and the <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state is `ACTIVE`.  During these actions, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> transitions its state to `NOT_READY` before completing its actions.  This should be regarded as a failure of the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span>.

![Responder Makes Unexpected State Change]({% link figures/mtconnect.png %} "Responder Makes Unexpected State Change"){: width="80.0%"}

Upon detecting an unexpected state change of the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span>, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> transitions its state to `FAIL`.  

The <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> resets any partial actions that were initiated and attempts to return to a condition where it is again ready to request a service.  If the recovery is successful, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> changes its state from `FAIL` to `READY`.  If for some reason the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> cannot return to a condition where it is again ready to request a service, it transitions its state from `FAIL` to `NOT_READY`.

Since the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> has failed to an invalid state, the condition of the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> is unknown.  Where possible, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> should try to reset to an initial state. 

The <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span>, as part of clearing the cause for the change to the unexpected state, should attempt to reset any partial actions that were initiated and then return to a condition where it is again ready to perform a service.  If the recovery is successful, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> changes its <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state from the unexpected state to `READY`.  If for some reason the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> is not again prepared to perform a service, it maintains its state as `NOT_READY`. 

### Responder or Requester Become UNAVAILABLE or Experience a Loss of Communication

In this scenario, a failure occurs in the communications connection between the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> and <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span>.  This failure may result from the [`InterfaceState`]({% link model/InterfaceInteractionModel/DataItemTypesforInterface/InterfaceState.md %}) from either piece of equipment returning a value of `UNAVAILABLE` or one of the pieces of equipment does not provide a heartbeat within the desired amount of time (See `Fundamentals` for details on heartbeat).   

![Requester - Responder Communication Failure 1]({% link figures/mtconnect.png %} "Requester - Responder Communication Failure 1"){: width="80.0%"}

![Requester - Responder Communication Failure 2]({% link figures/mtconnect.png %} "Requester - Responder Communication Failure 2"){: width="80.0%"}

When one of these situations occurs, each piece of equipment assumes that there has been a failure of the other piece of equipment.

When normal communications are re-established, neither piece of equipment should assume that the <span class="hoverterm">request and response<span class="definition" data-term="request and response">communications pattern that supports the transfer of information between an *agent* and a *client*. </span></span> state of the other piece of equipment remains valid.  Both pieces of equipment should set their state to `FAIL`.

The <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span>, as part of clearing its `FAIL` state, resets any partial actions that were initiated and attempts to return to a condition where it is again ready to request a service.  If the recovery is successful, the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> changes its state from `FAIL` to `READY`.  If for some reason the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> cannot return to a condition where it is again ready to request a service, it transitions its state from `FAIL` to `NOT_READY`.

The <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span>, as part of clearing its `FAIL` state, resets any partial actions that were initiated and attempts to return to a condition where it is again ready to perform a service.  If the recovery is successful, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> changes its <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> state from `FAIL` to `READY`.  If for some reason the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> is not again prepared to perform a service, it transitions its state from `FAIL` to `NOT_READY`.
