# Interface Interaction Model Package

In many manufacturing processes, multiple pieces of equipment must work together to perform a task.  The traditional method for coordinating the activities between individual pieces of equipment is to connect them using a series of wires to communicate equipment states and demands for action.  These interactions use simple binary ON/OFF signals to accomplished their intention.

In the MTConnect Standard, <span class="hoverterm">interfaces<span class="definition" data-term="interface">means by which communication is achieved between independent systems.</span></span> provides a means to replace this traditional method for interconnecting pieces of equipment with a structured <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> that provides a rich set of information used to coordinate the actions between pieces of equipment.  Implementers may utilize the information provided by this data model to (1) realize the interaction between pieces of equipment and (2) to extend the functionality of the equipment to improve the overall performance of the manufacturing process. 

The <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> used to implement <span class="hoverterm">interfaces<span class="definition" data-term="interface">means by which communication is achieved between independent systems.</span></span> provides a lightweight and efficient protocol, simplifies failure recovery scenarios, and defines a structure for implementing a Plug-And-Play relationship between pieces of equipment.  By standardizing the information exchange using this higher-level semantic information model, an implementer may more readily replace a piece of equipment in a manufacturing system with any other piece of equipment capable of providing similar <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> functions.

Two primary functions are required to implement the <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> for an <span class="hoverterm">interfaces<span class="definition" data-term="interface">means by which communication is achieved between independent systems.</span></span> and manage the flow of information between pieces of equipment.  Each piece of equipment needs to have the following:

* An <span class="hoverterm">agent<span class="definition" data-term="agent">software that collects data published from one or more piece(s) of equipment, organizes that data in a structured manner, and responds to requests for data from client software systems by providing a structured response in the form of a *response document* that is constructed using the *semantic data model* of a Standard. 
</span></span> which provides:
    
 - The data required to implement the <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span>.
    
 - Any other data from a piece of equipment needed to implement the <span class="hoverterm">interface<span class="definition" data-term="interface">means by which communication is achieved between independent systems.</span></span> – operating states of the equipment, position information, execution modes, process information, etc. 
 
* A client software application that enables the piece of equipment to acquire and interpret information from another piece of equipment.

## Request and Response Information Exchange

The [`DataItem`]({% link model/DeviceInformationModel/DataItems/DataItem.md %}) elements defined by the <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> each have a `REQUEST` and `RESPONSE` subtype.  These subtypes identify if the data item represents a <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> or a <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span>.  Using these data items, a piece of equipment changes the state of its <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> or <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> to indicate information that can be read by the other piece of equipment.  To aid in understanding how the <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> functions, one can view this <span class="hoverterm">interaction model<span class="definition" data-term="interaction model">model that defines how information is exchanged across an *interface* to enable interactions between independent systems.</span></span> as a simple state machine. 

The interaction between two pieces of equipment can be described as follows.  When the <span class="hoverterm">requester<span class="definition" data-term="requester">entity that initiates a *request* for information in a communications exchange.</span></span> wants an activity to be performed, it transitions its <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> state from a `READY` state to an `ACTIVE` state.  In turn, when the client on the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> reads this information and interprets the <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span>, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> announces that it is performing the requested task by changing its response state to `ACTIVE`.  When the action is finished, the <span class="hoverterm">responder<span class="definition" data-term="responder">entity that responds to a *request* for information in a communications exchange.</span></span> changes its response state to `COMPLETE`.  This pattern of <span class="hoverterm">request<span class="definition" data-term="request">communications method where a *client* transmits a message to an *agent*.  That message instructs the *agent* to respond with specific information.</span></span> and <span class="hoverterm">response<span class="definition" data-term="response">response *interface* which responds to a *request*.</span></span> provides the basis for the coordination of actions between pieces of equipment.  These actions are implemented using `EVENT` category data items.  (See `DataItem Types for Interface` for details on the [`Event`]({% link model/DeviceInformationModel/DataItems/DataItemTypes/Event.md %}) type data items defined for <span class="hoverterm">interfaces<span class="definition" data-term="interface">means by which communication is achieved between independent systems.</span></span>.)
